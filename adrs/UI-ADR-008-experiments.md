# UI-ADR-008: Experiments & A/B Testing

## Status
Accepted

## Dependencies
- **ADR-010**: Evaluation & Metrics System (ExperimentRunner, A/B testing infrastructure)
- **ADR-009**: Use Case Scenarios (scenarios to compare)
- **ADR-011**: Simulation Runtime & Scheduling (determinism, seed management)
- **UI-ADR-001**: Design System (comparison views, charts)
- **UI-ADR-007**: Results Analysis (per-variant results)

## Context

A/B testing is a **core use case for research workflows**:

| User Persona | Experiment Usage |
|--------------|-----------------|
| **Dr. Maya** (Researcher) | Daily - Systematic experiments comparing agent parameters, topologies, prompts |
| **Alex** (PM) | Weekly - Compare product variants, messaging strategies |
| **Sam** (Data Engineer) | Occasional - Test data generation quality across configurations |
| **Jordan** (Developer) | Debugging - Compare behavior across model versions |

### Requirements from UI Spec

1. **Experiment Builder**: Define variants with configuration changes
2. **Batch Execution**: Run multiple simulations with controlled variables
3. **Statistical Comparison**: Side-by-side results with significance testing
4. **Cost Estimation**: Pre-run cost preview
5. **Variant Diffing**: Visual comparison of configurations

### Current Gap

ADR-010 defines `ExperimentRunner` backend capability, but no UI specification exists for building, executing, and analyzing experiments.

## Decision

Implement a comprehensive **Experiment Management System** with visual experiment builder, batch execution dashboard, and statistical comparison tools.

## Consequences

### Positive

- **Systematic experimentation** supports rigorous research workflows
- **Statistical analysis** provides confidence in findings
- **Cost estimation** prevents budget surprises
- **Variant diffing** ensures controlled experiments
- **Reproducibility** via seed management enables verification

### Negative

- **Complex interface** may overwhelm casual users (mitigate with templates)
- **Statistical concepts** require user understanding (mitigate with explanations)
- **Long-running experiments** need background execution support

### Implementation Notes

1. **ExperimentRunner**: Use ADR-010 `ExperimentRunner` backend
2. **Statistics**: Use scipy.stats for ANOVA, effect sizes
3. **Progress Tracking**: WebSocket updates during execution
4. **Cost Estimation**: Based on historical token usage patterns
